# DECISIONS (решения и почему)

Этот файл фиксирует согласованные решения, чтобы не потерять контекст между сессиями.

## 2026-03-27 — Коммуникация и процесс

- **Режим работы**: я делаю проект полностью, вы утверждаете ключевые решения.
- **Что / Зачем / Важность — перед правками проекта и перед «запусти в терминале»**: три блока кратко давать (1) **что** именно предлагается сделать, (2) **зачем**, (3) **какая важность** — **непосредственно перед** тем, как ассистент **меняет файлы в репозитории**, и **так же** перед тем, как просим вас выполнить **конкретные команды в терминале** (например `docker compose …`, `pytest`, скрипты сборки): чтобы вы понимали смысл и не копировали «вслепую». В **обычном** разговоре без правок и без таких команд — отвечать **нормальным текстом**, без этого шаблона. Если ассистент снова нарушает это правило — достаточно одним словом написать **«DECISIONS»** (напоминание вернуться к этому пункту).
- **Контекст и память**: история чата ограничена; источник правды между сессиями — **`SESSION_STATE.md`**, **`DECISIONS.md`**, при смене темы деплоя — **`ИНСТРУКЦИЯ.md`**. При «обнулении» контекста ассистент опирается на эти файлы, а не только на последние реплики.
- **Тестирование**: тесты пишем **по ходу разработки** + финальный регресс перед релизом.
- **Запуск автотестов в Docker**: перед `run tests` всегда делаем `build tests`, чтобы не запускать устаревший образ.
- **Docker у ассистента нет доступа к сокету** — пересборку и `docker compose` выполняете **вы** в своём терминале. Каждый раз, когда после правок нужно **пересобрать/поднять бота или прогнать тесты в Docker**, ассистент в том же ответе (после блока **Что / Зачем / Важность**, если это команды) даёт **готовые строки** для копирования: `cd` в папку проекта + нужные `docker compose …` (как в `ИНСТРУКЦИЯ.md`). **Сразу под ними** — вторая команда для проверки лога бота, например `docker compose logs --tail=80 bot` (число строк — по ситуации: после ошибки можно больше). Вы вставляете обе по очереди; ассистент при необходимости читает **хвост** файла терминала Cursor или вы присылаете вывод — так подтверждаем, что сборка/старт прошли нормально.
- **Continuity**: добавлены `SESSION_STATE.md` и `DECISIONS.md`; по вашему сигналу завершения дня я обновляю их автоматически.
- **UX-стратегия**: сначала делаем аудит навигации/переходов (кнопки, back/main, FSM), и только потом просим вас о ручном прогоне. Это экономит ваше время и снижает количество мелких багов на проверке.
- **Текущий режим продакшна**: запускаем без домена, через polling; переход на webhook отложен.
- **CI**: GitHub Actions запускает `pytest` на ветках `main` / `master` / `develop` и на PR в эти ветки.
- **Режим Plan в Cursor и этот проект**: если в интерфейсе включён Plan mode, это **не** означает «только текст плана без правок». Ассистент **может и должен** по необходимости менять код, конфиги, тесты, миграции и прочие файлы репозитория — планирование и реализация здесь **не разводятся**: делаем то, что нужно для задачи, пока вы явно не попросили остановиться или ограничиться только планом.
- **Где лежит код сейчас vs передача заказчику**: текущая рабочая папка проекта у разработчика — **полный** набор всего нужного для работы (как удобно вам). Отдельную **«чистую» папку или архив без Git для брата/заказчика** собираем **в конце**, когда будет акт передачи — скриптами [`scripts/export_handover_directory.sh`](scripts/export_handover_directory.sh) (папка) и [`scripts/package_for_handover.sh`](scripts/package_for_handover.sh) (`.tar.gz`), см. `HANDOVER_CHECKLIST.md` §6.

## 2026-03-27 — Polling и FSM

- **`start_polling(..., handle_as_tasks=False)`**: апдейты обрабатываются **последовательно**, чтобы не было гонок при чтении/записи FSM в Redis (два callback подряд не затирали друг друга, в т.ч. WHO-5). Для небольшого бота это приемлемо; если снова включить параллельные задачи — нужны блокировки или атомарные обновления состояния.

## 2026-03-28 — Polling «для людей» и передача комплекта

- **Пользователи в Telegram**: по ощущениям **почти нет разницы** между polling и webhook — пишут боту как обычно, ответы приходят так же.
- **Заказчик / владелец**: **плюс** polling — не нужны домен и SSL, проще и дешевле запуск на VPS или на своём ПК. **Минус в быту**: процесс бота должен **постоянно работать** на какой-то машине (как и при webhook: без работающего процесса бот молчит). Для нашего масштаба **нет существенного минуса именно для пользователей**; редкие корпоративные сети с жёсткой блокировкой исходящих соединений — крайний случай, не типичный сценарий для закрытого бота на VPS.
- **Handover-папка**: собирается скриптом [`scripts/export_handover_directory.sh`](scripts/export_handover_directory.sh) в каталог **на уровень выше** проекта (`../Psy for IT BOT-YYYYMMDD` — то же имя, что у проекта, плюс дата; удобно для Windows-получателя). В комплект **не** кладём файл `.env` с секретами — токен и прочее передаются **отдельно** по безопасному каналу; в handover остаются только `.env.example` / `.env.prod.example`.
