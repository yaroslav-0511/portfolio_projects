# HANDOVER CHECKLIST (передача проекта)

## 1) Что должно быть готово технически
- [ ] `APP_ENV=prod` в `.env` на VPS
- [ ] `INVITE_RATE_LIMIT_ENABLED=true` в `.env` на VPS
- [ ] Бот запущен и стабилен: `docker compose logs --tail=150 bot` без traceback
- [ ] Миграции применяются автоматически при старте без ошибок
- [ ] Тесты пройдены на актуальном образе `tests` (`build tests` + `run --rm tests`)

## 2) Минимальная приемка перед передачей
- [ ] Owner: создаёт компанию, генерирует инвайт-коды
- [ ] User: проходит онбординг по коду и открывает главное меню
- [ ] Consultant: имеет доступ к меню консультанта
- [ ] Удаление/восстановление консультанта работает
- [ ] Перенос сессии пользователем работает
- [ ] User: полезные материалы и WHO-5 (опрос + сохранение); Owner: статистика WHO-5 (см. `CHECKLIST_U_C_O.md`)

## 3) Что передать брату
- [ ] **`ЧИТАТЬ_СНАЧАЛА.md`** в корне handover-папки (копия [`docs/START_HERE_RECIPIENT.md`](docs/START_HERE_RECIPIENT.md) — кладётся скриптом [`scripts/export_handover_directory.sh`](scripts/export_handover_directory.sh))
- [ ] Текущая `ИНСТРУКЦИЯ.md` (базовая инструкция)
- [ ] `DEPLOY_VPS.md` (операционный запуск и обновления)
- [ ] `HANDOVER_CHECKLIST.md` (этот чеклист)
- [ ] `CHECKLIST_U_C_O.md` (как вручную прогонять сценарии)
- [ ] Доступы: username бота, Owner Telegram ID, путь к VPS-проекту (когда появится)

## 4) Что НЕ передавать и где хранить
- [ ] Не отправлять публично `.env` и `BOT_TOKEN`
- [ ] Хранить `BOT_TOKEN` и доступы к VPS в приватном менеджере паролей
- [ ] Регулярно делать SQL-бэкап (`pg_dump`) и хранить копию вне VPS

## 5) Режим эксплуатации (текущий)
- [ ] Работаем в **polling** (домен не нужен)
- [ ] Обновление релиза: `docker compose up --build -d`
- [ ] Диагностика: `docker compose logs --tail=200 bot`

## 6) Финальная папка / архив для заказчика (в конце, когда отдаём проект)
Рабочая копия у разработчика может быть любой (включая Git). **Перед передачей брату** соберите отдельный комплект без служебной истории и без секретов.

- [ ] Убедиться, что в комплект **не** попадает файл `.env` с реальными токенами (передаётся заказчику отдельно, по безопасному каналу)
- [ ] В комплекте есть `.env.example` / `.env.prod.example` — заказчик копирует в `.env` и заполняет сам
- [ ] Исключить каталог **`.git`**, если заказчик просил «просто папку без Git»
- [ ] Прогнать тесты на актуальном коде (`build tests` + `run --rm tests`) перед упаковкой

**Автоматически (Linux / WSL / macOS):** из корня проекта.

Папка без Git (удобно открыть в проводнике / отдать как каталог), с **`ЧИТАТЬ_СНАЧАЛА.md`** в корне:
```bash
bash scripts/export_handover_directory.sh
```
Создаётся `../Psy for IT BOT-YYYYMMDD` рядом с каталогом проекта. Другой путь (если нужно своё имя или каталог): `bash scripts/export_handover_directory.sh "/path/to/Psy for IT BOT-YYYYMMDD"`

Архив `.tar.gz`:
```bash
bash scripts/package_for_handover.sh
```
Создаётся `../<имя_папки>-handover-YYYYMMDD.tar.gz` рядом с каталогом проекта. При необходимости свой путь: `bash scripts/package_for_handover.sh /path/to/out.tar.gz`

**Вручную:** скопировать папку проекта, удалить в копии `.git` и `.env`, проверить состав; в корень копии положить текст из [`docs/START_HERE_RECIPIENT.md`](docs/START_HERE_RECIPIENT.md) как `ЧИТАТЬ_СНАЧАЛА.md`.
